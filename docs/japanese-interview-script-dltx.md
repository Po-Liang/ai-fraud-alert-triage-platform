# Japanese Interview Script: AI Insurance Claims Review Copilot

この資料は、第一ライフテクノクロス株式会社の `AIエンジニア【社内AI戦略推進】` 面接で、このプロジェクトを自然に説明するための日本語トークスクリプトです。

前提として、このデモは PoC / 面接用デモです。実際のOCR、実顧客データ、実保険ルール、本番の支払い判断は含みません。AIは審査担当者を支援するものであり、支払い可否の最終判断は人間が行います。

## 使い方

面接では、この資料を丸暗記する必要はありません。まず30秒紹介を自然に言えるようにし、その後に画面を見せながら3分デモスクリプトの流れで説明するのがおすすめです。

特に伝えたいポイントは、単にコードを書いたことではなく、保険業務の課題をAI活用テーマに落とし込み、PoCとして形にし、RAG・書類処理・ガバナンス・本番化の論点まで考えていることです。

## 30秒紹介

このプロジェクトは、生命保険の保険金・給付金審査を想定した `AI Insurance Claims Review Copilot` というPoCデモです。

OCRで抽出されたような請求文書テキストを入力すると、請求種別や入院期間、診断名、提出書類などを整理し、審査担当者向けの要約と確認チェックリストを表示します。さらに、社内ガイダンスを検索するRAG機能により、回答と参照元を提示します。

ただし、AIが支払い可否を判断するのではなく、人間の審査担当者が原本書類、契約内容、社内ルールを確認して最終判断する設計にしています。

## 60秒紹介

このデモは、保険金・給付金の審査支援をテーマにした PoC です。第一ライフテクノクロス様のように、生命保険領域の業務システムや社内AI活用を支える会社であれば、書類処理、ナレッジ検索、審査支援はAI活用テーマとして相性が良いと考えました。

画面では、OCR出力を想定した請求文書テキストを入力し、バックエンドAPIに送信します。API側では、ルールベースのロジックで申請者名、請求種別、入院期間、診断名、提出書類などを抽出し、審査担当者向けの要約とレビュー用チェックリストを返します。

また、RAG型の社内ナレッジ検索として、架空の社内ガイダンス文書から関連する情報を検索し、回答と参照元を表示します。これにより、審査担当者が「なぜその回答になったのか」を確認しやすくしています。

構成としては、Reactのフロントエンド、API Gateway / Lambda のバックエンドAPI、RAG用サービス、そしてSecrets Managerを使った安全なAPIキー管理を意識しています。あくまでAIは業務判断を支援する位置づけで、最終的な支払い判断は人間が行う Human-in-the-loop 設計です。

## 3分デモスクリプト

### 1. フロントエンドを開く

「こちらが面接用に作成した `AI Insurance Claims Review Copilot` のデモ画面です。保険金・給付金の審査担当者が、請求書類の確認や社内ガイダンス検索を効率化する場面を想定しています。」

「このデモでは本物のOCRや実顧客データは使っていません。OCRで抽出された後のようなテキストを入力として扱い、あくまでPoCとして業務イメージを見せる構成にしています。」

### 2. サンプル請求テキストを説明する

「左側にあるテキストは、入院給付金の請求を想定した架空データです。申請者、請求種別、入院期間、診断名、提出書類が文章として含まれています。」

「本番ではここにAI-OCRや文書処理システムから取得したテキストを渡すイメージです。」

### 3. Analyze Claim をクリックする

「では `Analyze Claim` を押して、請求文書の内容を整理します。」

### 4. 抽出項目・要約・チェックリストを説明する

「結果として、請求種別、申請者名、入院期間、診断名、提出書類などが構造化されます。」

「下には、審査担当者向けの要約と確認チェックリストを表示しています。例えば、原本書類との照合、契約内容との確認、提出書類の不足確認など、担当者が次に見るべきポイントを整理します。」

「ここで重要なのは、AIが承認・否認を自動判断するのではなく、審査担当者が確認しやすい形に情報を整えることです。」

### 5. RAG質問を入力する

「次に、右側のRAG Copilotで社内ナレッジ検索を行います。例えば『入院給付金の審査で確認すべき項目は何ですか？』と質問します。」

「Ask Copilot を押すと、架空の社内ガイダンス文書から関連する情報を検索し、回答を返します。」

### 6. 回答と参照元を説明する

「回答だけでなく、参照元のガイダンス名やセクションも表示しています。これは、AIの回答をそのまま信じるのではなく、審査担当者が根拠を確認できるようにするためです。」

「本番化する場合は、ここをベクトル検索、文書アクセス制御、監査ログ、回答品質評価と組み合わせることで、より安全に運用できると考えています。」

### 7. AIガバナンスで締める

「最後に、このデモでは Human-in-the-loop を前提にしています。AIは要約、抽出、ナレッジ検索を支援しますが、支払い可否の最終判断は人間が行います。」

「原本書類、契約内容、社内ルールを確認したうえで判断する設計にしており、AI活用とガバナンスのバランスを意識しました。」

## 職務内容との対応

### AI活用テーマの企画・構想策定

保険金・給付金審査という業務課題に対して、書類確認、社内ナレッジ検索、審査チェックリスト化という具体的なAI活用テーマに分解しています。現場の作業負荷、確認漏れ、判断根拠の説明といった課題から、まず小さく検証できるPoCに落とし込む流れを意識しています。

### PoC

実データや本番業務に直接接続せず、架空データとローカルガイダンスを使って、業務価値と技術実現性を短時間で確認できるPoCとして構成しています。

### LLMアプリケーション開発

RAG型の質問応答、生成AIによる説明生成、Secrets Managerを使ったAPIキー管理を意識しています。現在はAWS構成ですが、Azure OpenAIへ移行する場合も、AI呼び出し部分をサービス層に閉じ込めているため差し替えやすい構成です。業務ロジックとAI呼び出しを分離することで、モデル変更や評価を行いやすくしています。

### AI-OCR / 書類処理

実OCRは未実装ですが、OCR出力のようなテキストを入力として扱い、請求情報の抽出、要約、確認リスト生成を行うことで、AI-OCR連携後の業務フローを表現しています。

### RAG / 社内ナレッジ検索

架空の社内ガイダンスを検索し、回答と参照元を返すRAG型業務支援を実装しています。回答に参照元を含めることで、利用者が根拠を確認できるようにしています。

### 保険金・給付金支払い / 審査支援AI

請求種別、入院期間、診断名、提出書類など、審査で確認される情報を整理し、担当者の確認作業を支援する形にしています。ただし、支払い可否の自動判断は行いません。

### AIガバナンス・セキュリティ

実顧客データを使わず、APIキーはSecrets Managerに保存し、フロントエンドには秘密情報を置かない設計です。また、AIが最終判断をしないことを画面とドキュメントで明示しています。RAG回答に参照元を出すことで、説明可能性と監査性も意識しています。

### MLOps / 運用高度化

現時点では本格的なMLOps基盤は未実装ですが、本番化する場合は、回答品質評価、プロンプト管理、モデル変更時の回帰テスト、監査ログ、CloudWatchメトリクス、アラートを追加する想定です。

## 面接で伝えたい専門性

このデモで一番伝えたいのは、単に生成AIを呼び出すだけではなく、業務課題、データの扱い、システム構成、ガバナンス、本番運用まで一連で考えられることです。

話すときは、次の3点を意識すると伝わりやすいです。

- 業務視点: 保険金・給付金審査における書類確認とナレッジ検索の負荷を下げる。
- 技術視点: OCR出力テキスト処理、RAG、バックエンドAPI、フロントエンドデモを分けて設計している。
- ガバナンス視点: AIは支援に限定し、人間が最終判断し、参照元や監査性を重視している。

## 想定質問と回答

### なぜこのユースケースを選びましたか？

保険業務では、書類確認、社内ルール確認、判断根拠の説明が重要だと考えました。第一ライフテクノクロス様のように生命保険領域の業務システムやAI活用を支える会社では、現場業務とAIをどう接続するかが重要だと思います。そこで、生成AIを単なるチャットではなく、AI-OCR、RAG、Human-in-the-loop と組み合わせた審査支援のPoCとして設計しました。

### OCRは実装していますか？

このデモでは実OCRは実装していません。現在はOCRで抽出された後のテキストを想定して入力しています。本番化する場合は、AWS Textract や Azure Document Intelligence などのOCRサービスと接続し、抽出結果をこのAPIに渡す構成にできます。今回の目的は、OCR後の業務支援フローを短時間で見せることです。

### Azure OpenAIに移行する場合はどうしますか？

AI呼び出し部分をサービス層に分離しているため、OpenAI APIの呼び出し部分を Azure OpenAI のエンドポイント、認証方式、デプロイ名に合わせて差し替えます。秘密情報はSecrets Manager相当のサービス、例えば Azure Key Vault に置き換える想定です。RAGや業務ロジックはできるだけ変更せず、AIプロバイダー依存の部分だけを差し替える方針です。

### 本番化する場合、何を追加しますか？

まず認証・認可、文書アクセス制御、監査ログ、入力データのマスキング、エラーハンドリングを強化します。RAGについては、ベクトルデータベース、文書更新パイプライン、回答品質評価、参照元の権限チェックを追加します。また、CloudWatchやX-Rayなどで運用監視も整備します。PoCで価値を確認した後、段階的に本番要件を満たしていく進め方が良いと考えています。

### AIの誤回答やハルシネーションにはどう対応しますか？

回答に参照元を必ず表示し、AIの回答を最終判断に使わない設計にします。また、プロンプトで「取得した文脈に基づいて回答する」ことを明示し、根拠が不足する場合は断定しないようにします。本番では、回答品質評価、ログ分析、専門家レビュー、NG回答パターンの検知も必要だと考えています。特に保険業務では、断定的な回答よりも、確認すべき観点を安全に提示することが重要だと思います。

### AIが支払い可否を判断しますか？

いいえ、判断しません。AIは情報整理、要約、チェックリスト作成、社内ナレッジ検索を支援するだけです。支払い可否の最終判断は、審査担当者が原本書類、契約内容、社内ルールを確認したうえで行う設計です。

### 効果測定はどのように行いますか？

PoC段階では、審査担当者の作業時間、確認漏れの削減、社内ガイダンス検索時間、回答満足度、参照元クリック率などを測定します。本番化前には、AI出力の正確性、担当者による修正率、業務プロセスへの影響も評価したいです。技術指標だけでなく、現場で本当に使いやすいかも確認します。

### MLOpsとして何を考慮しますか？

モデルやプロンプトのバージョン管理、評価データセット、回帰テスト、ログ監視、品質メトリクス、コスト監視、障害時のフォールバックを考慮します。RAGでは、文書更新時のインデックス再作成、検索品質評価、古い文書が回答に使われない仕組みも重要です。

### セキュリティ面で何を意識しましたか？

フロントエンドにAPIキーやAWS認証情報を置かないこと、APIキーをSecrets Managerで管理すること、実顧客データを使わないこと、AIが最終判断をしないことを意識しました。本番では、認証・認可、暗号化、監査ログ、アクセス制御、データマスキングをさらに強化します。

### なぜRAGの回答に参照元を含めていますか？

審査業務では、回答の内容だけでなく根拠が重要だからです。参照元を表示することで、担当者が社内ルールやガイダンスを確認でき、AIの回答を検証しやすくなります。これはハルシネーション対策や説明責任にもつながります。

## Short English Summary

This interview script explains the project as an insurance claim review copilot demo. The demo shows OCR-output-like text analysis, structured claim extraction, review checklist generation, and RAG-based internal guidance search. It is positioned as a PoC, not a production system.

The key message is that AI supports human reviewers but does not make final payment decisions. The candidate should emphasize business fit, secure secret handling, source-backed RAG answers, human-in-the-loop governance, and future production improvements such as authentication, access control, audit logs, vector search, evaluation, and MLOps.
